# 部署与容量配置指南

本文面向知识库维护者，覆盖一键部署、国内镜像回退、Windows 原生依赖安装、10–20 人使用时的并发配置、域名/内网穿透和图谱关联检索。

## 1. 推荐部署路径与配置向导

生产或项目组共享环境优先使用 Docker Compose；Windows 原生模式用于源码调试和个人开发。

| 场景 | 命令 | 说明 |
|---|---|---|
| Windows Docker | `.\start.ps1 up` | 首次进入配置向导，生成 `.env` 后拉取模型并等待健康 |
| Linux/macOS/WSL | `sh ./start.sh up` | 与 Windows Docker 启动器行为一致 |
| Windows 原生开发 | `.\start-dev.ps1` | 自动安装/启动 Python、Ollama、Memurai、Chroma、MinIO 和 Gateway |
| 修改 Docker 配置 | `.\start.ps1 configure` / `sh ./start.sh configure` | 按类别重新配置并更新 `.env`；Windows 也可双击 `scripts\init-config.bat` |

首次执行不带部署参数的 `up` 时，启动器会依次询问硬件档位、GPU、镜像源、访问方式、数据目录、Embedding 模型和管理员账号，并生成包含随机密钥的 `.env`。访问方式为复选项，可同时勾选“仅本机、局域网、公网、Cloudflare Tunnel”；确认后，“具体配置”只会继续询问所选方式需要的名称、域名和 Token。以后执行 `up`、`down`、`status` 或 `logs` 都会自动复用已保存的 GPU、镜像源和全部访问方式。

需要调整配置时执行：

```powershell
# Windows；也可以双击 scripts\init-config.bat
.\start.ps1 configure
```

```bash
# Linux / macOS / WSL
sh ./start.sh configure
# 已安装 make 时也可以执行 make configure
```

重配菜单可单独修改硬件/GPU、镜像源、访问方式、数据/模型或初始管理员。访问方式使用 `↑/↓` 移动，按 `Space` 勾选或取消，按 `Enter` 提交；只有不支持逐键读取的终端才回退到逗号分隔的编号输入。修改完成后再次执行 `up` 才会重建或重启受影响的容器。首次向导中途退出时，`DEPLOY_CONFIGURED` 会保持 `false`，下次 `up` 将继续完整配置。初始管理员配置只在账号库尚未创建账号时生效，已有管理员应在后台修改密码。

`init` 是非交互初始化入口，显式参数和 `-NonInteractive` / `--non-interactive` 适合 CI、远程脚本和无人值守安装：

```powershell
.\start.ps1 init -NonInteractive -Profile minimum -Gpu cpu -Source official
```

```bash
sh ./start.sh init --non-interactive --profile minimum --cpu --source official
```

首次 Docker 启动后访问 `http://<服务器IP>/admin`。Docker 只向局域网开放 Nginx 的 80/443；Redis、Chroma、MinIO、Ollama 和 Gateway 直连端口都绑定在 `127.0.0.1`。

### 1.1 注册全局 `knowbase` 命令

交互式 `up` 在部署成功后会询问是否注册全局命令。无人值守部署可以直接指定：

```powershell
# Windows 当前用户 PATH；无需管理员权限
.\start.ps1 up -InstallCli
# 已完成部署时也可单独注册
.\start.ps1 cli-install
```

```bash
# Linux 默认安装到 ~/.local/bin
sh ./start.sh up --install-cli
# 已完成部署时也可单独注册
sh ./start.sh cli-install
```

重新打开终端后即可从任意目录运行：

```bash
knowbase health
knowbase gateway restart
knowbase gateway logs --tail 100
knowbase configure
knowbase doctor
```

`gateway restart` 只重启 Gateway 容器，不会重启 Redis、Chroma、MinIO 或 Ollama，并会等待 `/health` 恢复后才返回。`knowbase restart` 则重启完整 Compose 服务。首次部署或容器尚未创建时必须使用 `knowbase up`。

Windows 默认把 `knowbase.cmd` 安装到已位于用户 PATH 的 `%LOCALAPPDATA%\Microsoft\WindowsApps`，项目绑定信息单独保存在 `%LOCALAPPDATA%\KnowledgeBaseManagement`，可避免 Windows Terminal 常驻进程继续使用旧 PATH。若精简系统不存在 `WindowsApps`，安装器才回退到独立目录并广播环境变化；原终端仍未识别时需关闭所有 Windows Terminal 窗口后重新打开。Linux 安装器会把受标记的 PATH 配置写入 `~/.profile` 及当前 Bash/Zsh 配置文件，卸载时只删除自己的标记块。可用 `knowbase cli status` 检查，使用 `knowbase cli uninstall` 卸载。项目整体移动后，在新目录重新执行 `cli-install` 即可更新绑定路径。

## 2. 国内镜像与自动回退

默认顺序如下：

| 类型 | 首选 | 自动回退 |
|---|---|---|
| Docker Hub 镜像 | `m.daocloud.io/docker.io/` | Docker Hub 官方镜像名 |
| Python 包 | 清华 PyPI `https://pypi.tuna.tsinghua.edu.cn/simple` | `https://pypi.org/simple` |
| Debian apt | 清华 Debian 镜像 | `deb.debian.org` |
| Windows Python 安装器 | 清华 Python 镜像 | Python.org |
| Windows MinIO | MinIO 中国下载地址 | MinIO 官方 Community 下载地址 |
| Ollama / Memurai | `winget` | 软件官方安装器/MSI |

向导选择“国内镜像优先”时，Docker 启动器第一次 `pull/build` 失败后会自动追加 `docker-compose.official.yml` 并完整重试；选择“官方源”时会直接使用官方源，不再做一次无意义的国内源尝试。选择结果保存在 `DEPLOY_IMAGE_SOURCE`，生命周期命令会自动复用。已在 `.env` 自定义 `MIRROR_PREFIX`、`PIP_INDEX_URL` 或 `APT_MIRROR` 时，国内源模式会先使用你的值。

镜像配置依据：[DaoCloud Public Image Mirror](https://docs.daocloud.io/en/community/mirror)、[清华 PyPI 镜像帮助](https://mirrors.tuna.tsinghua.edu.cn/help/pypi/)。

## 3. 硬件档位与容量

配置向导或显式启动参数会把 `deploy/profiles/<档位>.env` 写入 `.env` 或 `.env.local`，并把选择记录到 `HARDWARE_PROFILE`。显式指定档位会更新已有配置；未指定时首次 Docker 配置默认建议 `recommended`，后续命令复用已保存的档位。

| 档位 | 硬件建议 | 典型使用 | 核心参数 |
|---|---|---|---|
| `minimum` | 4 核 CPU、8 GB 内存、SSD，可无独显 | 1–5 人、开发/低频查询 | 搜索并发 4、Ollama 并行 1、图谱 1 跳 |
| `recommended` | 8 核 CPU、16 GB 内存、SSD；建议 6 GB+ NVIDIA 显存 | 10–20 人间歇使用、最多约 12 个同时执行的检索 | 搜索并发 12、Ollama 并行 4、图谱 2 跳 |
| `high-performance` | 12 核+、32 GB+、NVMe；建议 12 GB+ NVIDIA 显存 | 10–20 个持续同时检索，或更大知识库 | 搜索并发 20、Ollama 并行 8、HTTP 连接 40 |

“用户数”不等于“同一毫秒内的检索请求数”。若 20 个客户端会持续并行发起向量查询，应使用高性能档位并做实际压测；CPU-only 环境通常先从 `minimum` 开始，避免 Ollama 因过度并行造成所有请求同时变慢。

```powershell
# Windows Docker
.\start.ps1 up -Profile recommended
.\start.ps1 up -Profile high-performance -Gpu gpu

# Windows 原生开发
.\start-dev.ps1 -Profile minimum
```

```bash
sh ./start.sh up --profile recommended
make up PROFILE=high-performance
```

## 4. 并发和超时配置

| 配置 | 作用 | 推荐调法 |
|---|---|---|
| `SEARCH_MAX_CONCURRENCY` | 单 Gateway 同时执行的搜索数 | 先按档位；出现 503 且资源有余量再逐步增加 |
| `SEARCH_QUEUE_TIMEOUT_MS` | 等待并发槽位的时间 | 突发流量可增至 3000–5000；交互优先则保持较短 |
| `SEARCH_TOTAL_TIMEOUT_MS` | 获得槽位后整次搜索时限 | CPU 模式可增加，高性能 GPU 可降低 |
| `SEARCH_VECTOR_TIMEOUT_MS` | Ollama + Chroma 向量通道时限 | 必须小于总时限，通常占最大预算 |
| `OLLAMA_NUM_PARALLEL` | Ollama 同时处理模型请求数 | 受显存/内存约束，不应盲目等于用户数 |
| `EMBEDDING_MAX_CONNECTIONS` | Gateway 到 Ollama 的连接池上限 | 应不小于搜索并发，通常为其 1–2 倍 |
| `EMBEDDING_MAX_BATCH_SIZE` | 入库时单批 Embedding 文本数 | 内存不足时降低；主要影响写入，不直接提高查询并发 |
| `RATE_LIMIT_DEFAULT` | 单 API Key 每分钟 HTTP 请求数 | 多 Agent 共用一个 Key 时需提高，最好为每个 Agent 单独发 Key |

过载时 Gateway 在队列超时后返回 HTTP 503，并给出 `retry_after_ms`；调用方应做带抖动的退避重试。某个检索阶段超时不会让其他通道结果丢失，响应中的 `status=degraded`、`retrieval_errors` 和 `timings_ms` 可用于定位瓶颈。

调优顺序：

1. 先观察 `/metrics`、`timings_ms`、Ollama 日志和 CPU/GPU/内存。
2. 若主要是排队 503，且 Ollama/内存仍有余量，再增加 `SEARCH_MAX_CONCURRENCY`。
3. 若向量通道超时，先降低 `OLLAMA_NUM_PARALLEL` 或启用 GPU，不要只增加 Gateway 并发。
4. 若上下文补充慢，降低 `SEARCH_CONTEXT_MAX_CHARS` 或调用时传 `include_context=false`。
5. 常规调整优先运行 `configure`；修改后重新执行 `up`。如需手工微调 `.env` 中的高级参数，应在最后调整，因为再次选择硬件档位会重写该档位管理的并发参数。

## 5. Windows 原生自动安装

`.\start-dev.ps1` 的执行顺序：

1. 查找 Python 3.11+；缺失时优先 `winget install Python.Python.3.13`，再按清华镜像、Python.org 回退静默安装。
2. 使用清华 PyPI 安装 `requirements.txt`；失败时切换官方 PyPI。
3. 查找/安装 Ollama，启动服务并最多重试三次拉取 `bge-m3`。
4. 对 `REDIS_URL` 执行真实 Redis `PING`；本地 Redis 不可用时查找/安装 Memurai Developer，优先启动 Windows 服务，失败后再尝试直接运行可执行文件。
5. 安装并启动 Chroma，下载并启动 MinIO，再拉起 Gateway。脚本每 10 秒提示一次等待进度，只有 Gateway `/health` 返回 200 才报告全部健康；超时会输出 Gateway 日志尾部并返回非零退出码。

系统安装可能触发 Windows UAC。公司受管控电脑可执行 `.\start-dev.ps1 -NoAutoInstall`，此时脚本只检查系统依赖并安装项目 Python 包；缺少 Python、Ollama 或 Memurai 时会明确停止。自动化或排障时可用 `.\start-dev.ps1 -InitOnly -Profile minimum` 只生成并验证 `.env.local`，不会创建数据目录或启动服务；用 `-Background` 可在 Gateway 健康后返回终端，后续执行 `-Stop` 停止本项目进程。

安装依据：[Python Windows 安装选项](https://docs.python.org/3/using/windows.html)、[Ollama Windows 下载](https://ollama.com/download/windows)、[Memurai 安装文档](https://docs.memurai.com/en/installation)。

## 6. 局域网、域名和内网穿透

四种访问方式可以组合启用，向导显示当前勾选状态：

```text
[ ] 1) 仅本机
[x] 2) 局域网
[x] 3) 公网
[x] 4) Cloudflare Tunnel
```

移动光标并用空格勾选第 2、3、4 项，再按回车提交后，向导会按“局域网 → 公网 → Cloudflare Tunnel”的顺序进入具体配置。选择结果写入 `DEPLOY_ACCESS_MODES=lan,domain,cloudflare`；兼容字段 `DEPLOY_ACCESS_MODE` 会写为 `hybrid`，`DEPLOY_TUNNEL_MODE` 则根据是否包含 Tunnel 自动维护。未勾选方式不会参与当前运行；公网域名和 Tunnel Hostname/Token 会继续保留，方便以后重新启用。

### 6.1 局域网

运行 `configure` 并勾选“局域网”后，向导会自动获取计算机名，并从带默认路由的活动网卡中检测有效 IPv4，过滤回环地址和 `169.254.x.x`。默认值形如 `KNOWLEDGE-SERVER,192.168.1.100`，直接回车即可让 Nginx 同时接受计算机名和 IP；有线、Wi-Fi、VPN 等产生多个候选时，可删除不需要的地址或补充自定义内网域名。客户端连接 `http://<计算机名或服务器IP>/mcp`；不要使用 Docker 的 `:8000` 直连端口。勾选局域网不会关闭公网或 Tunnel，两者是否启用完全由同一组复选项决定。

### 6.2 有公网 IP 的域名

1. 将域名 A/AAAA 记录指向公网 IP；动态地址可在后台“设置 → DDNS”配置 Provider。
2. 将路由器 80/443 转发到服务器，并放行主机防火墙/云安全组。
3. 放置 `nginx/ssl/<域名>/fullchain.pem` 和 `nginx/ssl/<域名>/privkey.pem`。
4. 运行 `configure`，勾选“公网”，并填写公网直连域名。若还需要局域网或 Tunnel，可在同一步一并勾选。以下是“局域网 + 公网”的示例：

```dotenv
DEPLOY_ACCESS_MODES=lan,domain
PUBLIC_DOMAIN=kb.example.com
EXTERNAL_DOMAIN=kb.example.com
INTERNAL_DOMAIN=kb.internal.example.com
CORS_ORIGINS=*
REVERSE_PROXY_ENABLED=true
```

5. 重启并验证 `https://kb.example.com/health` 和 `https://kb.example.com/mcp`。

### 6.3 无公网 IP：Cloudflare Tunnel

1. 在 Cloudflare 控制台创建 remotely-managed tunnel。
2. 新建 Public Hostname，将 Service 配为 `http://nginx:80`。这个主机名只在 Compose 网络内解析。
3. 运行配置向导，勾选“Cloudflare Tunnel”，填写 Public Hostname 和 Tunnel Token。输入 Token 时终端不会回显，配置摘要也只显示是否已配置：

```powershell
.\start.ps1 configure
.\start.ps1 up
```

```bash
sh ./start.sh configure
sh ./start.sh up
```

Tunnel 作为可选 Compose profile 运行，只建立出站连接，不要求家庭/办公网络开放入站端口。模式同时记录在 `DEPLOY_ACCESS_MODES` 和兼容字段 `DEPLOY_TUNNEL_MODE`，所以后续执行 `logs`、`status` 或 `down` 不必重复传 `--tunnel` / `-Tunnel`。Token 通过 `TUNNEL_TOKEN` 环境变量传给容器，不写进命令行。Cloudflare 官方建议 Docker 使用 remotely-managed tunnel，命令和 Token 参数说明见 [Set up Cloudflare Tunnel](https://developers.cloudflare.com/tunnel/setup/) 与 [Run parameters](https://developers.cloudflare.com/tunnel/advanced/run-parameters/)。

### 6.4 同时启用公网直连与 Tunnel

公网直连域名保存在 `PUBLIC_DOMAIN`，Tunnel 主机名保存在 `CLOUDFLARE_PUBLIC_HOSTNAME`。两者应使用不同名称，例如：

```dotenv
DEPLOY_ACCESS_MODES=lan,domain,cloudflare
PUBLIC_DOMAIN=kb.example.com
CLOUDFLARE_PUBLIC_HOSTNAME=tunnel.kb.example.com
EXTERNAL_DOMAIN=kb.example.com
DEPLOY_ACCESS_MODE=hybrid
DEPLOY_TUNNEL_MODE=cloudflare
CORS_ORIGINS=*
```

`EXTERNAL_DOMAIN` 是兼容 Nginx 和旧版本的运行字段：启用公网直连时优先使用 `PUBLIC_DOMAIN`，否则在仅启用 Tunnel 时使用 Tunnel Hostname。局域网模式为了适配内部 Demo 客户端会保持 `CORS_ORIGINS=*`；未启用局域网时，向导会把已选公网/Tunnel 的 HTTPS Origin 组合写入 CORS。Cloudflare Public Hostname 的 Service 始终指向 `http://nginx:80`，公网直连仍通过宿主机 80/443 和本地证书工作，因此两条入口可并行使用。

旧 `.env` 只有 `DEPLOY_ACCESS_MODE` 时会自动迁移：原 `domain` 保留为 `lan,domain`，原 `cloudflare` 保留为 `local,cloudflare`。显式使用 `-Tunnel cloudflare` / `--tunnel cloudflare` 会在已有组合中增加 Tunnel，使用 `off` 只移除 Tunnel，不会关闭其他入口。

公网或穿透模式务必修改初始管理员密码、为每个 Agent 创建最小 `read` scope Key、限制可访问目录、将 `CORS_ORIGINS` 收紧到实际 HTTPS 域名，并在外部网关配置身份策略。不要暴露 6379、8001、9000/9001 或 11434。

## 7. 图谱关联检索

旧版本图谱只生成可视化文件，不参与 `search_knowledge`。当前版本会同时生成 `kbdata/graph/retrieval_index.json`，并在基础混合召回后扩展关联文档：

- `co_tag`：共享标签，标签越多边权越高。
- `declares_core_entity`：文档与其头部声明核心实体的关系。
- `co_entity`：共享核心实体的文档关系，优先级高于共享标签。
- `same_directory`：同目录结构关系。
- `semantically_similar`：可选语义边，保存实际余弦相似度。

核心实体由 Markdown 头部规则提取，不调用 LLM：`核心实体：`、`实体：`、`Core Entities:`、`Entities:` 都可使用；标签同样兼容 `标签：`、`Tags:`。批量写入、编辑或重索引后，请按批次重建图谱以刷新这些关系。

为避免一个通用标签或大目录产生平方级海量边，50 篇以内的关系组保留完整连接；更大的组使用两跳可达的稀疏星形连接。检索侧只保留高权重前沿，防止高连接度图谱拖慢并发搜索。

```json
{
  "semantic_threshold": 0.72
}
```

可在后台构建，也可调用 MCP `build_knowledge_graph`。建议从 `0.72` 开始：误召回多则调高，关联召回不足则小幅调低。文档增删改后图谱不会在每次写入时做昂贵的全量重建，应按批次或定时重新构建。

| 配置 | 推荐值 | 说明 |
|---|---|---|
| `GRAPH_RETRIEVAL_ENABLED` | `true` | 关闭后完全不读取图谱 |
| `GRAPH_RETRIEVAL_WEIGHT` | `0.35` | 控制图谱候选进入最终排序的力度 |
| `GRAPH_RETRIEVAL_MAX_RESULTS` | `3` | 单次最多引入的关联文档数 |
| `GRAPH_RETRIEVAL_MAX_HOPS` | `2` | 最大关系跳数，最多限制为 3 |
| `GRAPH_RETRIEVAL_SEED_COUNT` | `3` | 用多少个强基础命中作为扩展起点 |
| `GRAPH_RETRIEVAL_MIN_EDGE_WEIGHT` | `0.25` | 过滤弱关系 |
| `GRAPH_RETRIEVAL_TIMEOUT_MS` | `800` | 超时后保留基础搜索结果并标记降级 |

验证时查看搜索结果的 `channel=graph` 和 `association_reason`。如果图谱尚未构建或文件不存在，搜索会自动退回向量、关键词和结构通道。

## 8. 上线检查

- 已运行 `configure` 检查配置摘要，访问模式、GPU、镜像源和 Tunnel 与预期一致。
- `start.ps1 status` / `start.sh status` 健康检查通过。
- 管理员初始密码已修改，`SESSION_SECRET` 与 MinIO 密码为随机长值。
- Agent Key 按人/按应用分配，只有需要写入的 Key 才有 `write` scope。
- `/metrics`、审计日志、备份和恢复流程已验证。
- 图谱已在当前文档批次后重建；搜索能看到预期的基础与图谱通道。
- 10–20 用户上线前用真实查询集做并发测试，不只测试 `/health`。
- 公网模式已启用 HTTPS/外部身份策略，内部存储端口没有公网暴露。
