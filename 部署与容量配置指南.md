# 部署与容量配置指南

本文面向知识库维护者，覆盖一键部署、国内镜像回退、Windows 原生依赖安装、10–20 人使用时的并发配置、域名/内网穿透和图谱关联检索。

## 1. 推荐部署路径

生产或项目组共享环境优先使用 Docker Compose；Windows 原生模式用于源码调试和个人开发。

| 场景 | 命令 | 说明 |
|---|---|---|
| Windows Docker | `.\start.ps1 up` | 首次创建 `.env`、生成密钥、应用推荐档位、拉取模型并等待健康 |
| Linux/macOS/WSL | `sh ./start.sh up` | 与 Windows Docker 启动器行为一致 |
| Windows 原生开发 | `.\start-dev.ps1` | 自动安装/启动 Python、Ollama、Memurai、Chroma、MinIO 和 Gateway |
| Cloudflare 穿透 | `.\start.ps1 up -Tunnel cloudflare` | 需先在 `.env` 填 `CLOUDFLARE_TUNNEL_TOKEN` |

首次 Docker 启动后访问 `http://<服务器IP>/admin`。Docker 只向局域网开放 Nginx 的 80/443；Redis、Chroma、MinIO、Ollama 和 Gateway 直连端口都绑定在 `127.0.0.1`。

## 2. 国内镜像与自动回退

默认顺序如下：

| 类型 | 首选 | 自动回退 |
|---|---|---|
| Docker Hub 镜像 | `m.daocloud.io/docker.io/` | Docker Hub 官方镜像名 |
| Python 包 | 清华 PyPI `https://pypi.tuna.tsinghua.edu.cn/simple` | `https://pypi.org/simple` |
| Debian apt | 清华 Debian 镜像 | `deb.debian.org` |
| Windows Python 安装器 | 清华 Python 镜像 | Python.org |
| Windows MinIO | MinIO 中国下载地址 | MinIO 官方 Community 下载地址 |
| Ollama / Memurai | `winget` | 软件官方安装器/MSI |

Docker 启动器第一次 `pull/build` 失败后会自动追加 `docker-compose.official.yml` 并重试。已在 `.env` 自定义 `MIRROR_PREFIX`、`PIP_INDEX_URL` 或 `APT_MIRROR` 时，启动器先使用你的值，失败后仍回退官方源。

镜像配置依据：[DaoCloud Public Image Mirror](https://docs.daocloud.io/en/community/mirror)、[清华 PyPI 镜像帮助](https://mirrors.tuna.tsinghua.edu.cn/help/pypi/)。

## 3. 硬件档位与容量

启动参数会把 `deploy/profiles/<档位>.env` 覆盖到 `.env` 或 `.env.local`。显式指定档位会更新已有配置；`auto` 只在首次创建配置时应用 `recommended`。

| 档位 | 硬件建议 | 典型使用 | 核心参数 |
|---|---|---|---|
| `minimum` | 4 核 CPU、8 GB 内存、SSD，可无独显 | 1–5 人、开发/低频查询 | 搜索并发 4、Ollama 并行 1、图谱 1 跳 |
| `recommended` | 8 核 CPU、16 GB 内存、SSD；建议 6 GB+ NVIDIA 显存 | 10–20 人间歇使用、最多约 12 个同时执行的检索 | 搜索并发 12、Ollama 并行 4、图谱 2 跳 |
| `high-performance` | 12 核+、32 GB+、NVMe；建议 12 GB+ NVIDIA 显存 | 10–20 个持续同时检索，或更大知识库 | 搜索并发 20、Ollama 并行 8、HTTP 连接 40 |

“用户数”不等于“同一毫秒内的检索请求数”。若 20 个客户端会持续并行发起向量查询，应使用高性能档位并做实际压测；CPU-only 环境通常先从 `minimum` 开始，避免 Ollama 因过度并行造成所有请求同时变慢。

```powershell
# Windows Docker
.\start.ps1 up -Profile recommended
.\start.ps1 up -Profile high-performance -Gpu gpu

# Windows 原生开发
.\start-dev.ps1 -Profile minimum
```

```bash
sh ./start.sh up --profile recommended
make up PROFILE=high-performance
```

## 4. 并发和超时配置

| 配置 | 作用 | 推荐调法 |
|---|---|---|
| `SEARCH_MAX_CONCURRENCY` | 单 Gateway 同时执行的搜索数 | 先按档位；出现 503 且资源有余量再逐步增加 |
| `SEARCH_QUEUE_TIMEOUT_MS` | 等待并发槽位的时间 | 突发流量可增至 3000–5000；交互优先则保持较短 |
| `SEARCH_TOTAL_TIMEOUT_MS` | 获得槽位后整次搜索时限 | CPU 模式可增加，高性能 GPU 可降低 |
| `SEARCH_VECTOR_TIMEOUT_MS` | Ollama + Chroma 向量通道时限 | 必须小于总时限，通常占最大预算 |
| `OLLAMA_NUM_PARALLEL` | Ollama 同时处理模型请求数 | 受显存/内存约束，不应盲目等于用户数 |
| `EMBEDDING_MAX_CONNECTIONS` | Gateway 到 Ollama 的连接池上限 | 应不小于搜索并发，通常为其 1–2 倍 |
| `EMBEDDING_MAX_BATCH_SIZE` | 入库时单批 Embedding 文本数 | 内存不足时降低；主要影响写入，不直接提高查询并发 |
| `RATE_LIMIT_DEFAULT` | 单 API Key 每分钟 HTTP 请求数 | 多 Agent 共用一个 Key 时需提高，最好为每个 Agent 单独发 Key |

过载时 Gateway 在队列超时后返回 HTTP 503，并给出 `retry_after_ms`；调用方应做带抖动的退避重试。某个检索阶段超时不会让其他通道结果丢失，响应中的 `status=degraded`、`retrieval_errors` 和 `timings_ms` 可用于定位瓶颈。

调优顺序：

1. 先观察 `/metrics`、`timings_ms`、Ollama 日志和 CPU/GPU/内存。
2. 若主要是排队 503，且 Ollama/内存仍有余量，再增加 `SEARCH_MAX_CONCURRENCY`。
3. 若向量通道超时，先降低 `OLLAMA_NUM_PARALLEL` 或启用 GPU，不要只增加 Gateway 并发。
4. 若上下文补充慢，降低 `SEARCH_CONTEXT_MAX_CHARS` 或调用时传 `include_context=false`。
5. 修改 `.env` 后重启服务；硬件档位会覆盖相关参数，请最后再做手工微调。

## 5. Windows 原生自动安装

`.\start-dev.ps1` 的执行顺序：

1. 查找 Python 3.11+；缺失时优先 `winget install Python.Python.3.13`，再按清华镜像、Python.org 回退静默安装。
2. 使用清华 PyPI 安装 `requirements.txt`；失败时切换官方 PyPI。
3. 查找/安装 Ollama，启动服务并最多重试三次拉取 `bge-m3`。
4. 查找/安装 Memurai Developer，作为 Windows 原生 Redis 兼容服务监听 6379。
5. 安装并启动 Chroma，下载并启动 MinIO，最后以前台热重载方式运行 Gateway。

系统安装可能触发 Windows UAC。公司受管控电脑可执行 `.\start-dev.ps1 -NoAutoInstall`，此时脚本只检查系统依赖并安装项目 Python 包；缺少 Python、Ollama 或 Memurai 时会明确停止。

安装依据：[Python Windows 安装选项](https://docs.python.org/3/using/windows.html)、[Ollama Windows 下载](https://ollama.com/download/windows)、[Memurai 安装文档](https://docs.memurai.com/en/installation)。

## 6. 局域网、域名和内网穿透

### 6.1 局域网

保持 `EXTERNAL_DOMAIN=`，把 `INTERNAL_DOMAIN` 设置为主机名、内网域名或服务器 IP。客户端连接 `http://<服务器IP>/mcp`；不要使用 Docker 的 `:8000` 直连端口。

### 6.2 有公网 IP 的域名

1. 将域名 A/AAAA 记录指向公网 IP；动态地址可在后台“设置 → DDNS”配置 Provider。
2. 将路由器 80/443 转发到服务器，并放行主机防火墙/云安全组。
3. 放置 `nginx/ssl/<域名>/fullchain.pem` 和 `nginx/ssl/<域名>/privkey.pem`。
4. 设置：

```dotenv
EXTERNAL_DOMAIN=kb.example.com
INTERNAL_DOMAIN=kb.internal.example.com
CORS_ORIGINS=https://kb.example.com
REVERSE_PROXY_ENABLED=true
```

5. 重启并验证 `https://kb.example.com/health` 和 `https://kb.example.com/mcp`。

### 6.3 无公网 IP：Cloudflare Tunnel

1. 在 Cloudflare 控制台创建 remotely-managed tunnel。
2. 新建 Public Hostname，将 Service 配为 `http://nginx:80`。这个主机名只在 Compose 网络内解析。
3. 只复制 Tunnel Token 到 `.env`：

```dotenv
CLOUDFLARE_TUNNEL_TOKEN=你的长Token
EXTERNAL_DOMAIN=kb.example.com
CORS_ORIGINS=https://kb.example.com
```

4. 启动：

```powershell
.\start.ps1 up -Tunnel cloudflare
.\start.ps1 logs -Tunnel cloudflare
```

```bash
sh ./start.sh up --tunnel cloudflare
sh ./start.sh logs --tunnel cloudflare
```

Tunnel 作为可选 Compose profile 运行，只建立出站连接，不要求家庭/办公网络开放入站端口。Token 通过 `TUNNEL_TOKEN` 环境变量传给容器，不写进命令行。Cloudflare 官方建议 Docker 使用 remotely-managed tunnel，命令和 Token 参数说明见 [Set up Cloudflare Tunnel](https://developers.cloudflare.com/tunnel/setup/) 与 [Run parameters](https://developers.cloudflare.com/tunnel/advanced/run-parameters/)。

公网或穿透模式务必修改初始管理员密码、为每个 Agent 创建最小 `read` scope Key、限制可访问目录、将 `CORS_ORIGINS` 收紧到实际 HTTPS 域名，并在外部网关配置身份策略。不要暴露 6379、8001、9000/9001 或 11434。

## 7. 图谱关联检索

旧版本图谱只生成可视化文件，不参与 `search_knowledge`。当前版本会同时生成 `kbdata/graph/retrieval_index.json`，并在基础混合召回后扩展关联文档：

- `co_tag`：共享标签，标签越多边权越高。
- `same_directory`：同目录结构关系。
- `semantically_similar`：可选语义边，保存实际余弦相似度。

为避免一个通用标签或大目录产生平方级海量边，50 篇以内的关系组保留完整连接；更大的组使用两跳可达的稀疏星形连接。检索侧只保留高权重前沿，防止高连接度图谱拖慢并发搜索。

```json
{
  "semantic_threshold": 0.72
}
```

可在后台构建，也可调用 MCP `build_knowledge_graph`。建议从 `0.72` 开始：误召回多则调高，关联召回不足则小幅调低。文档增删改后图谱不会在每次写入时做昂贵的全量重建，应按批次或定时重新构建。

| 配置 | 推荐值 | 说明 |
|---|---|---|
| `GRAPH_RETRIEVAL_ENABLED` | `true` | 关闭后完全不读取图谱 |
| `GRAPH_RETRIEVAL_WEIGHT` | `0.35` | 控制图谱候选进入最终排序的力度 |
| `GRAPH_RETRIEVAL_MAX_RESULTS` | `3` | 单次最多引入的关联文档数 |
| `GRAPH_RETRIEVAL_MAX_HOPS` | `2` | 最大关系跳数，最多限制为 3 |
| `GRAPH_RETRIEVAL_SEED_COUNT` | `3` | 用多少个强基础命中作为扩展起点 |
| `GRAPH_RETRIEVAL_MIN_EDGE_WEIGHT` | `0.25` | 过滤弱关系 |
| `GRAPH_RETRIEVAL_TIMEOUT_MS` | `800` | 超时后保留基础搜索结果并标记降级 |

验证时查看搜索结果的 `channel=graph` 和 `association_reason`。如果图谱尚未构建或文件不存在，搜索会自动退回向量、关键词和结构通道。

## 8. 上线检查

- `start.ps1 status` / `start.sh status` 健康检查通过。
- 管理员初始密码已修改，`SESSION_SECRET` 与 MinIO 密码为随机长值。
- Agent Key 按人/按应用分配，只有需要写入的 Key 才有 `write` scope。
- `/metrics`、审计日志、备份和恢复流程已验证。
- 图谱已在当前文档批次后重建；搜索能看到预期的基础与图谱通道。
- 10–20 用户上线前用真实查询集做并发测试，不只测试 `/health`。
- 公网模式已启用 HTTPS/外部身份策略，内部存储端口没有公网暴露。
