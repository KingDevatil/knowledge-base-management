# Agent 使用指南 — 中央知识库 MCP 操作手册

本文档面向已接入中央知识库 MCP 的 AI Agent，说明如何有效使用知识库提供的工具进行检索、阅读和管理文档。

---

## 1. 工具总览

知识库暴露以下 MCP 工具，分为三类：

### 检索工具（read 权限）

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `search_knowledge` | 向量检索，找到与查询语义最匹配的文档片段 | 用户提问、需要查阅资料时 |
| `get_document` | 获取某篇文档的完整内容+所有切片 | 找到相关文档后、需要全文阅读时 |
| `list_documents` | 按条件列出文档（支持分页、标签/路径筛选） | 浏览目录、发现有哪些文档时 |
| `list_directories` | 列出知识库的目录树结构 | 了解知识组织方式、确定搜索范围时 |

### 写入工具（write 权限）

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `add_document` | 添加新文档 | 需要向知识库写入新内容时 |
| `update_document` | 更新已有文档 | 文档内容需要修正或补充时 |
| `delete_document` | 删除文档 | 文档过期或错误时 |

### 维护工具（write 权限）

| 工具 | 用途 | 调用时机 |
|------|------|---------|
| `rename_directory` | 重命名目录，自动移动该目录下所有文档 | 目录结构调整时 |
| `delete_directory` | 删除目录，该目录下文档移回根目录 | 目录废弃时 |
| `reindex_document` | 用当前切片策略重建某文档的切片和向量 | 切片配置变更后需要刷新现有文档时 |

## 2. 检索最佳实践

### 2.1 搜索查询技巧

`search_knowledge` 基于向量语义匹配，不是关键词匹配。这意味着：

- **用自然语言提问**效果优于关键词堆砌
- **描述具体场景**比抽象概念更精准
- **一次检索不够**可以换个角度重新搜索

```
✅ 好的查询："项目的用户认证流程中，JWT Token 的刷新机制是怎样的？"
❌ 不好的查询："Token 认证"
```

### 2.2 组合筛选

善用 `filter_tags` 和 `filter_path` 缩小检索范围：

```
search_knowledge(query="登录流程", filter_tags=["技术"], filter_path="系统资料")
```

### 2.3 多轮检索模式

复杂问题时采用"先探索→再聚焦"的策略：

```
第1轮：search_knowledge(query="大致了解项目有哪些方面的资料", top_k=10)
      → 了解知识库覆盖的领域

第2轮：list_directories()
      → 查看目录结构，找到对应方向

第3轮：search_knowledge(query="具体问题详细描述", filter_path="对应目录")
      → 在目标目录下深入检索
```

### 2.4 调整结果数量

`top_k` 默认为 5，可根据需求调整：

- 快速浏览：`top_k=3`
- 全面调研：`top_k=10`
- 穷举搜索：`top_k=20`

## 3. 文档阅读流程

从检索到完整阅读的推荐流程：

```
步骤1: search_knowledge(query="...", top_k=5)
       → 找到相关文档片段，获得 doc_id

步骤2: get_document(doc_id="xxx")
       → 获取该文档的完整内容和所有切片

步骤3: 阅读全文后回答用户，或继续搜索相关内容
```

> 注意：`search_knowledge` 返回的是**切片级别**的结果（文档的一部分），需要完整上下文时务必调用 `get_document` 获取全文。

## 4. 文档写入规范

### 4.1 添加文档

```json
{
  "title": "API 接口文档",
  "content": "# API 接口规范\n\n## 认证\n...",
  "path": "技术文档/API参考",
  "tags": ["技术", "API"]
}
```

- `title` 是文档的唯一标识名，建议清晰且唯一
- `content` 为 Markdown 格式，建议用 `##` / `###` 分节
- `path` 对应目录树中的路径，多级用 `/` 分隔
- `tags` 推荐 2-4 个标签，便于后续筛选

### 4.2 更新文档

更新时会做**变更检测**，内容无变化时自动跳过。需要提供完整的 `title` 和 `content`（覆盖式更新）。

```json
{
  "doc_id": "已有的doc_id",
  "title": "API 接口文档",
  "content": "# 更新后的完整内容..."
}
```

### 4.3 写入目录结构建议

```
项目资料/
  系统设计/       ← 架构设计、技术方案
  接口规范/       ← API 定义、协议说明
  运营手册/       ← 操作流程、应急预案
技术文档/
  API参考/
  开发指南/
运维/
  部署文档/
  监控告警/
```

## 5. 典型场景示例

### 场景一：用户问了一个专业问题

```
用户："生产环境的 Redis 主从切换流程是怎样的？"

你的处理：
1. search_knowledge(query="Redis 主从切换的详细步骤和注意事项", top_k=5)
   → 找到相关片段和对应的文档

2. 如果片段信息足够 → 直接回答
   如果片段信息不足 → get_document(doc_id) 获取全文后回答
```

### 场景二：用户让总结某个方向的所有资料

```
用户："帮我整理一下项目里所有关于接口规范相关的资料"

你的处理：
1. list_directories() → 找到"接口规范"目录
2. list_documents(path="项目资料/接口规范") → 列出该目录下所有文档
3. 逐个 get_document 获取全文
4. 综合所有内容给出总结
```

### 场景三：用户让补充新知识

```
用户："把这篇新写的部署方案加到知识库里"

你的处理：
1. add_document(title="生产环境部署方案", content=内容, path="运维/部署文档", tags=["运维", "部署"])
```

### 场景四：用户发现文档过时

```
用户："XX文档里的信息已经过时了，更新一下"

你的处理：
1. search_knowledge(query="XX文档相关的内容") → 找到 doc_id
2. get_document(doc_id) → 获取当前内容
3. update_document(doc_id=xxx, title=..., content=更新后的内容)
```

## 6. 限制与注意事项

- **写入有锁保护**：`add_document` / `update_document` 需获取分布式写锁，并发写入时可能返回 423（写锁被占用），等待几秒后重试即可
- **Content Hash 变更检测**：`update_document` 会对比内容 SHA256，未变化时跳过写入
- **切片策略**：文档以 `###` 标题为边界分段，每段不超过 512 字符（可配置）
- **Embedding 模型**：使用 Ollama + bge-m3，中文效果优秀
- **doc_id 不可变**：文档的 `doc_id` 在创建后不可更改，`update_document` 不能改 ID

## 7. 快速参考

### 目录和标签筛选

```json
// 检索时只搜索"系统设计"目录
search_knowledge(query="缓存策略", filter_path="项目资料/系统设计")

// 检索时只搜索带"技术"标签的文档
search_knowledge(query="部署流程", filter_tags=["技术"])

// 组合筛选
search_knowledge(query="API 鉴权方式", filter_tags=["接口"], filter_path="项目资料/接口规范")
```

### 文档分页

```json
// 第1页，每页10条
list_documents(path="运维/部署文档", limit=10, offset=0)

// 第2页
list_documents(path="运维/部署文档", limit=10, offset=10)
```

### 错误处理

| 响应 | 含义 | 处理方式 |
|------|------|---------|
| `423 Locked` | 写锁被占用 | 等待几秒后重试 |
| `503` | Embedding 服务不可用 | 检查 Ollama 是否运行 |
| `401` | API Key 无效 | 检查 Key 是否过期或填错 |
| `404` | 文档不存在 | 文档可能已被删除 |
